## Fix 1: Wireframe mesh never written (Response 8)

**Symptom:** Wireframe cube appeared as a flat triangle (the template.glb placeholder geometry).

**Root cause:** In `ShapeManager.Update()`, the `_wireframeMeshWritten` flag was set to `true` unconditionally, even when `WireframeManager.TryWriteMesh()` returned early because the mesh wasn't ready yet. The wireframe geometry was never actually written.

**Fix:** Changed `TryWriteMesh()` from `void` to `bool`, returning `false` when the mesh isn't ready and `true` after a successful write. Updated `ShapeManager` to only set the flag on success:
```csharp
_wireframeMeshWritten = _wireframe.TryWriteMesh();
```

**Files:** `WireframeManager.cs`, `ShapeManager.cs`

## Fix 2: Shape/label meshes re-written every frame (Response 8)

**Symptom:** App crashed when pinching to spawn a shape.

**Root cause:** `TryWriteShapeMesh()` and `TryWriteLabelMesh()` were called every frame for every shape using the resize overload of `WriteMeshBuffers`. This re-allocated and re-wrote static geometry every update cycle, which could destabilize the native runtime.

**Fix:** Added `ShapeMeshWritten` and `LabelMeshWritten` bool flags to `SpawnedShape`. Each mesh is now written exactly once after it becomes ready:
```csharp
if (!shape.ShapeMeshWritten)
    shape.ShapeMeshWritten = TryWriteShapeMesh(shape);
```

Also changed `TryWriteShapeMesh` and `TryWriteLabelMesh` from `void` to `bool` to report success. Added try-catch around `SpawnShape` for crash resilience.

**Files:** `SpawnedShape.cs`, `ShapeManager.cs`, `LabelManager.cs`

## Fix 3: template.glb missing TEXCOORD_0 (Response 9)

**Symptom:** Wireframe still showed as triangle; shapes still crashed on spawn â€” fixes 1 and 2 alone didn't resolve the issue.

**Root cause:** The `template.glb` generated by the `trimesh` Python library did **not include the `TEXCOORD_0` attribute** (UVs). The ShapeSpawner's `MeshResource` declared UV buffer descriptors (`VaMeshBufferTypeExt.VertexTexcoord0`), but the GLB lacked that data. The native layer couldn't properly set up those buffers, causing:
- `WriteMeshBuffers` to fail silently or crash
- The original template triangle to remain visible (since mesh writes never succeeded)

**Verification:** Inspected the GLB JSON and found only `POSITION` and `NORMAL` â€” no `TEXCOORD_0`. Compared against the working `BoxTextured.glb` used by SpinningCube, which has all three attributes.

**Fix:** Rewrote `tools/generate_template.py` to construct the GLB manually using raw `struct.pack` and JSON (no trimesh dependency). The new template includes all 4 buffer types:
- Indices (uint32)
- POSITION (vec3)
- NORMAL (vec3)
- TEXCOORD_0 (vec2)

Material named `"mat"` with white base color, metallic=0, roughness=1.

New file: 1244 bytes (was 1112 bytes without UVs).

**Files:** `tools/generate_template.py`, `assets/shapes/template.glb`

## Fix 4: Unsafe list cast and error handling (Response 9)

**Symptom:** Potential crash from cast; no error visibility.

**Root cause:** `ShapeSpawnerVolume.HandleOnUpdate` cast `_shapeManager.ActiveShapes` (type `IReadOnlyList<SpawnedShape>`) to `List<SpawnedShape>`. While it worked because the backing type was `List`, it was fragile. Also, any unhandled exception in `HandleOnUpdate` or `ShapeManager.Update` would crash the entire app with no diagnostic output.

**Fix:**
- Changed `HandInteractionManager.Update()` and `CheckPokes()` to accept `IReadOnlyList<SpawnedShape>` instead of `List<SpawnedShape>`
- Changed `LabelManager.UpdateBillboards()` to accept `IReadOnlyList<SpawnedShape>`
- Removed the cast in `ShapeSpawnerVolume`
- Added try-catch in `HandleOnUpdate` and `ShapeManager.Update` with `Console.Error` logging

**Files:** `HandInteractionManager.cs`, `LabelManager.cs`, `ShapeSpawnerVolume.cs`, `ShapeManager.cs`

## Fix 5: Template too minimal + buffer descriptor mismatch + decoupleAccessors (Response 10)

**Symptom:** All shapes still appeared as triangles; wireframe still not rendering correctly.

**Root causes (3 combined issues):**

1. **template.glb too minimal** â€” The template contained a single triangle (3 vertices, 3 indices). The native mesh resize infrastructure may not reliably resize from such a minimal starting point. All working SDK samples that use `WriteMeshBuffers` with resize start from a fully formed mesh (e.g., Simple.gltf = 24-vertex cube, scene.glb = full model).

2. **Buffer descriptor mismatch** â€” `WireframeManager` declared only 3 buffer descriptors (Index, Position, Normal) but the template.glb contained 4 attributes (including TEXCOORD_0). This mismatch between the MeshResource descriptor list and the template's actual attributes could cause the native layer to fail to set up the buffers correctly for resize operations.

3. **`decoupleAccessors: false`** â€” The C++ MeshEdit sample (the only SDK sample that demonstrably resizes meshes with the same pattern) uses `decoupleAccessors: true`. With `false`, accessor buffers are shared, which can interfere with resize operations that need to reallocate buffer memory.

**Fixes:**

1. Regenerated `template.glb` as a full unit cube (24 vertices, 36 indices) instead of a single triangle. This matches the mesh complexity of `Simple.gltf` and `BoxTextured.glb` used by working samples.

2. Added `VaMeshBufferTypeExt.VertexTexcoord0` to `WireframeManager.Descriptors` so all MeshResource instances now declare all 4 buffer types matching the template's attributes.

3. Changed `decoupleAccessors` from `false` to `true` in all three MeshResource constructors (WireframeManager, ShapeManager, LabelManager).

4. Added `WriteMeshBuffers` return value checking â€” previously the return value was ignored, so failures were silent. Now logs to `Console.Error` if the write fails.

**Files:** `generate_template.py`, `template.glb` (regenerated), `WireframeManager.cs`, `ShapeManager.cs`, `LabelManager.cs`

## Fix 8: Wireframe rewritten as 12 scaled template cubes (Response 13)

**Symptom:** Wireframe still showed as a solid cyan cube â€” `WriteMeshBuffers` resize overload was silently failing.

**Root cause:** The `WriteMeshBuffers` resize overload is unreliable for replacing geometry on a template.glb. Multiple attempts (decoupleAccessors toggling, SetVisible toggling, buffer descriptor matching) all failed. The only working SDK sample (VolumetricMusicPlayer) uses a custom scene.glb with a pre-built plane mesh that exactly matches the target topology â€” not a generic template.

**Fix:** Completely eliminated `MeshResource` and `WriteMeshBuffers` from the wireframe. Instead, the wireframe cube is built from **12 `VisualElement` instances**, each sharing the same `ModelResource` (template.glb unit cube). Each element uses `SetScale(VaVector3f)` with non-uniform scale to stretch the cube into a thin beam:
- 4 edges along X: scale `(VolumeSize, WireThickness, WireThickness)`
- 4 edges along Y: scale `(WireThickness, VolumeSize, WireThickness)`
- 4 edges along Z: scale `(WireThickness, WireThickness, VolumeSize)`

`TryWriteMesh()` now returns `true` immediately â€” no mesh writing needed.

**Files:** `WireframeManager.cs`

## Fix 9: Labels rewritten as flat scaled cubes (Response 14)

**Symptom:** Labels were huge â€” the template cube was being shown at full size because `WriteMeshBuffers` was failing for label meshes too.

**Root cause:** Same as Fix 8 â€” the `WriteMeshBuffers` resize overload doesn't work reliably with the template.glb.

**Fix:** Eliminated `MeshResource` and `WriteMeshBuffers` from labels entirely. Labels now use the template cube with non-uniform scale to create a flat plane:
- X = `LabelWidth` (0.0762m / 3 inches)
- Y = `LabelHeight` (0.0254m / 1 inch)
- Z = 0.001m (paper-thin)

Removed `LabelMesh` and `LabelMeshWritten` from `SpawnedShape`. Animation no longer waits for label mesh. New `SetLabelScale()` helper applies the non-uniform scale during animation.

**Files:** `LabelManager.cs`, `ShapeManager.cs`, `SpawnedShape.cs`

## Fix 10: Label text upside-down (Response 15)

**Symptom:** Text on label was rendered upside-down.

**Root cause:** The template cube's UV mapping on the visible face has V coordinates that run opposite to the expected direction. When the cube is squashed into a plane, the texture appears inverted vertically.

**Fix:** Negated the Y component of the label scale (`y = -(t * Constants.LabelHeight)`). This flips the cube along Y, which flips the UV mapping and renders the text right-side up.

**Files:** `ShapeManager.cs`

## Fix 11: Shapes all rendered as cubes (Response 16)

**Symptom:** All spawned shapes appeared as cubes regardless of the label saying "Sphere", "Cylinder", etc.

**Root cause:** `WriteMeshBuffers` with the resize overload was silently failing for shapes too (same issue as wireframe and labels). The template.glb cube geometry was never being replaced at runtime, so every shape was a cube.

**Fix:** Completely eliminated `WriteMeshBuffers` from shapes. Created a Python script (`tools/generate_shapes.py`) that generates 5 pre-built GLB files â€” `cube.glb`, `sphere.glb`, `cylinder.glb`, `cone.glb`, `pyramid.glb` â€” each containing the correct unit-sized mesh geometry with POSITION, NORMAL, TEXCOORD_0 and a material named "mat".

`ShapeManager.SpawnShapeCore()` now loads the shape-specific GLB directly:
```csharp
var shapeUri = VolumetricApp.GetAssetUri($"{shapeName.ToLowerInvariant()}.glb");
```

Removed `MeshResource`, `ShapeDescriptors`, `TryWriteShapeMesh()`, `CopyToBuffer()`, and `ShapeMeshWritten` tracking entirely. Shapes render their correct geometry as soon as the GLB loads â€” no runtime mesh replacement needed.

**Files:** `tools/generate_shapes.py` (new), `assets/shapes/*.glb` (5 new), `CsShapeSpawner.csproj`, `ShapeManager.cs`, `SpawnedShape.cs`
